# Структура на кода и PR процес (при ревю)

> Изведено от повтарящи се бележки в ревютата. Допълва [`../AGENTS.md`](../AGENTS.md) и
> [`../CONTRIBUTING.md`](../CONTRIBUTING.md) с това, което ревюто изисква на практика.

## Структура на кода и преизползване

- **Преизползвайте съществуващите компоненти — „нищо ново за поддръжка".** Градете страниците от
  `ShareBar`, `DataTable`, `TotalsStrip`; собствените графики следват `SankeyDiagram` (чисто SVG,
  без външна библиотека). Нов компонент или зависимост иска изрична обосновка.
- **Целият SQL живее в `@sigma/db` с тестове** — loader-ите викат експортирани функции; без суров
  SQL в route файловете. Нова аналитика делегира към съществуващите параметризирани заявки.
- **Изнасяйте споделена логика в `app/lib` / `app/components`** — без дублиране между сродни
  страници (`authority.tsx` ↔ `company.tsx`).
- **Пазете doc-коментарите в `@sigma/api-contract` в синхрон със SQL-а** — остарял коментар на DTO
  поле вече е подвеждал контрибутори. Остарелите коментари са дефекти.

## PR процес

- **Една логическа промяна на PR**; несвързани поправки → отделен issue. Чист refactor не бива да
  крие промени в поведението — изнесете ги в отделен commit и ги опишете.
- **Rebase върху последния `main` преди merge** — при движение на `main` ребейзвайте; конфликтни
  PR-и не се мърджват.
- **Не връщайте умишлени поправки назад** — ако промяната ви докосва предишно решение (отбелязано в
  commit/коментар), изрично отбележете преоткриването и предложете резолюция.
- **Координирайте реда на merge при споделени файлове** — когато PR-ът ви докосва същите редове/
  файлове като друг отворен PR (`app.css`, `wrangler.jsonc`, `pnpm-lock.yaml`, `workers/*`),
  отбележете го в двата PR-а и обявете реда; вторият мърджър чисти (тихите семантични merge-ове
  чупят CI без конфликтен маркер).
- **Големи функционалности влизат като един дълготраен споделен branch**, не като много PR-а, всеки
  пипащ обща инфраструктура.
- **Всеки PR се връзва към проследяващо issue** (триажна стъпка); това е счетоводство, не критика —
  не го бъркайте с искане за промяна.

## Конвенции

- Решенията по дизайн и границите на обхвата се записват в `docs/`, не в коментари по кода или в
  описанието на PR-а.
- Текстът към потребителя е на български.
- Conventional commits: `<type>(<scope>): <subject>`, subject малки букви, повелително, без точка
  накрая. **Никога `Co-Authored-By:` trailer** — CI може да го grep-ва.
